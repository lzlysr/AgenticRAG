# AgenticRAG 项目说明

本文是面向面试审查的项目级说明，目标不是只写“用了 RAG”，而是把项目从数据、索引、Agent 推理、评测、SFT、GRPO 训练到外部框架的协作关系讲清楚。

我把阅读范围分成两层：

- 自研核心层：`agents/`、`retrieval/`、`llm/`、`api/`、`evaluation/`、`scripts/`、`training/`、`config.py`、`requirements*.txt`、数据资产。这部分是项目真正的业务逻辑和面试重点，下面逐模块、逐文件、逐函数说明。
- 外部框架层：`LLaMA-Factory/` 和 `verl/` 是完整拷入的训练框架。项目通过配置、数据格式转换、工具类和奖励函数接入它们，不应在面试中把这两个大型框架的通用实现说成自己的核心贡献。

## 二、项目一句话定位

这个项目是在金融多跳问答场景中构建一个 Agentic RAG 系统。它不只是“向量检索加生成”，而是把问题拆成多步子任务，由 Agent 自主选择检索工具、收集证据、验证证据是否充分，再合成答案。项目还配套了金融多跳 QA 合成、SFT 行为克隆、GRPO 多轮工具强化学习、离线评测和消融实验。

面试时可以这样概括：

> 我做的是一个金融多跳 QA 的 Agentic RAG 复现和训练系统。推理时用 LangGraph 编排 Router、Planner、Executor、Verifier、Synthesizer。检索层同时支持 BM25、FAISS+BGE-M3、知识图谱检索和 RRF 融合。训练上先用 oracle trace 做 ReAct SFT，再用 verl 做多轮工具调用 GRPO，奖励函数同时优化答案正确性、证据忠实性、检索 hop 命中和工具格式。

## 五、LLM 统一调用层

### `llm/client.py`

这是所有 LLM 调用的适配层。项目约定所有模型都走 OpenAI 兼容协议，所以本地 vLLM 和云端 API 都可以挂到同一个接口。

核心对象和函数：

- `ModelConfig`：保存一个模型的 base URL、实际模型名、API key、max_len、temperature、top_p、top_k、timeout、retry 次数等。
- `ModelConfig.client`：懒加载 `OpenAI` client，首次访问才创建连接。
- `MODEL_CONFIGS`：模型注册表，默认有 `Qwen3-4B`、`Qwen3-32B`、`gpt-oss-120b`。面试时可以强调“扩展模型不改业务代码，只在这里注册”。
- `LLMStats`：线程安全统计 agent/judge 调用次数和总延迟。
- `_strip_think(text)`：去掉 Qwen 推理模型可能输出的 `<think>...</think>`。
- `_format_messages(messages)`：把纯字符串 prompt 转成 OpenAI messages。
- `get_from_llm(messages, model_name, **kwargs)`：核心调用函数，负责校验模型名、套用默认参数、调用 chat completion、失败重试。
- `_extract_json(text)`：从模型输出中提取 JSON，兼容 markdown code block、文本中夹杂 JSON 的情况。
- `agent_chat()` / `agent_chat_json()`：Agent 推理侧的文本或 JSON 调用。
- `judge_chat()` / `judge_chat_json()`：Judge 侧调用。
- `get_from_ks_openai()`：旧脚本兼容别名。

项目里 Router、Planner、Verifier、Synthesizer、数据合成、Judge、翻译、别名增强等都最终走这里。

## 六、Agent 编排层

### 总体设计

`agents/` 使用 LangGraph 的 `StateGraph` 实现一个 PEV 循环：

- P: Planner，把复杂问题拆成子查询计划。
- E: Executor，执行检索工具并收集 evidence。
- V: Verifier，判断证据是否足够，不够则反馈给 Planner 重规划。
- 最后 Synthesizer 基于 evidence 输出短答案。

### `agents/state.py`

定义 `AgentState(TypedDict)`，是 LangGraph 每个节点读写的状态。

关键字段：

- `query`：原始问题。
- `query_type`：`simple` 或 `multi_hop`。
- `plan`：子任务列表，每个元素包含 `id`、`sub_query`、`tool`、`depends_on`、`status`。
- `current_step`：当前执行到第几个 plan step。
- `evidence`：累积证据，使用 `Annotated[list[dict], operator.add]` 让 LangGraph 合并列表。
- `tool_calls`：工具调用日志。
- `verification_result`：`sufficient`、`insufficient` 或 `contradiction`。
- `verification_feedback`：重规划指导。
- `final_answer`：最终答案。
- `iteration_count`：Planner/Verifier 循环轮数。
- `total_tool_calls`：总工具调用次数。
- `trace`：完整执行轨迹。

### `agents/router.py`

- `route_query(state)`：调用 `agent_chat_json()`，用 router prompt 判断问题是单跳还是多跳。模型输出不可信时默认走 `multi_hop`，这是偏保守设计。
- `route_decision(state)`：LangGraph 条件边函数，根据 `query_type` 返回 `simple` 或 `multi_hop`。

### `agents/planner.py`

- `TOOL_DESCRIPTIONS`：给 LLM 的工具说明，包括 keyword、semantic、read_chunk、graph。
- `plan(state)`：生成或重生成检索计划。

`plan()` 的逻辑：

1. 读取原始 query 和当前 iteration。
2. 如果是重规划，收集已有 evidence 摘要，把 verifier feedback 拼入 prompt。
3. 从 executor 的 `TOOL_REGISTRY` 动态读取当前可用工具。这样消融实验可以禁用某些工具。
4. 调用 planner prompt，要求 LLM 返回 JSON array。
5. 如果模型输出不可解析，则兜底为单步 `semantic_search`。
6. 给每个 step 加 `status="pending"`。
7. 返回新的 `plan`、`current_step=0`、`iteration_count+1` 和 trace。

面试重点：Planner 不是固定规则，而是模型根据问题和工具说明动态选择工具，并支持重新规划。

### `agents/executor.py`

这是工具执行器。

- `TOOL_REGISTRY`：当前启用工具。
- `_ALL_TOOLS`：完整工具备份，供消融实验恢复。
- `_ensure_tools()`：懒加载检索工具，把 `keyword_search`、`semantic_search`、`read_chunk`、`graph_search` 注册进工具表。
- `_normalize_tool(tool_field)`：把 planner 可能输出的 `tool` 字段统一成工具名列表。支持：
  - 单工具字符串：`"semantic_search"`。
  - 多工具列表：`["keyword_search", "semantic_search"]`。
  - `"hybrid_search"` 标记，从 step 的 `tools` 字段读取具体工具列表。
- `execute_step(state)`：按计划执行检索。
- `should_continue_executing(state)`：如果 plan 执行完或工具预算耗尽就去 verifier，否则继续 executor。

`execute_step()` 的关键行为：

1. 检查依赖 `depends_on` 是否已完成。
2. 标准化工具字段。
3. 如果是多工具或 hybrid，过滤无效工具后调用 `retrieval.hybrid_search.multi_tool_search()`。
4. 如果是单工具，直接从 `TOOL_REGISTRY` 调用。
5. 多步执行时会把上一轮 evidence 的 top 文本片段加入后续 sub_query 的 context。
6. 每步最多保留 top 5 results 到 evidence。
7. 更新 `total_tool_calls`、`tool_calls` 和 trace。

### `agents/verifier.py`

- `_get_chunk_ids(evidence_list)`：从 evidence 中收集所有 chunk_id。
- `verify(state)`：把已收集 evidence 拼成文本，调用 verifier prompt 判断是否足以回答原问题。
- `after_verification(state)`：LangGraph 条件边，决定去 synthesize 还是 replan。

一个重要细节是 evidence 去重检测：如果重规划后的新 evidence 和旧 evidence 的 chunk_id 重叠超过 `EVIDENCE_OVERLAP_THRESHOLD=0.9`，即使 verifier 还说 insufficient，也强制 sufficient。这个设计是为了防止模型在相同证据上反复重规划、陷入循环。

### `agents/synthesizer.py`

- `_extract_short_answer(text)`：从 LLM 输出中提取短答案。支持 `<answer>...</answer>`，去掉 “Answer:” 等前缀，过长时取第一句。
- `synthesize(state)`：把 evidence 中每个 step 的 top 3 chunk 拼成上下文，调用 synthesizer prompt 输出最终答案，再抽取短答案。
- `simple_rag(state)`：简单问题直通路径，只调用 `semantic_search()` 一次，然后让 LLM 基于 top 3 context 回答。

### `agents/graph.py`

- `build_graph(enable_verifier=True, enabled_tools=None)`：组装 LangGraph。
- `run_query(query, **kwargs)`：构造初始 state 并执行图。
- `get_default_app()`：默认图单例。

`build_graph()` 支持消融实验：

- `enable_verifier=False` 可以关闭 verifier，让 executor 后直接 synthesize。
- `enabled_tools=[...]` 可以只启用部分工具。

图结构：

```text
router
  -> simple_rag -> END
  -> planner -> executor
       -> executor 继续执行下一步
       -> verifier
            -> planner 重规划
            -> synthesizer -> END
```

### `agents/prompts.py`

这个文件管理所有中英文 prompt 和模型能力 profile。

- 小模型 profile：7B/8B/14B 使用更宽松、更简化的 planner/verifier，`max_iterations=3`，`max_retrieval_calls=10`。
- 大模型 profile：其他模型使用更严格的 planner/verifier，`max_iterations=5`，`max_retrieval_calls=15`。
- `PROMPT_LANG="zh"` 时使用中文 prompt。
- `get_profile(model_name=None)`：根据模型名和语言返回 profile。

面试可以强调：项目不是一套 prompt 到处用，而是按模型大小和语言切换 prompt，并把检索预算也放在 profile 里。

## 七、检索层

### 总体检索工具

检索工具都返回统一结构：

```python
{
    "chunk_id": "...",
    "text": "...",
    "title": "...",
    "score": 0.0,
    "source": "..."
}
```

Executor 不关心底层是 BM25、FAISS 还是 KG，只按这个结构收 evidence。

### `retrieval/embedder.py`

- `_get_model(device=None)`：按设备懒加载 BGE-M3。用 `_models` 和 `_locks` 做 device 级缓存和线程锁。
- `encode(texts, batch_size=64, device=None)`：编码文本并归一化，返回 `np.float32` 矩阵。

设计点：同一设备只加载一次模型，避免多线程重复占显存。

### `retrieval/reranker.py`

- `_get_model(device=None)`：按设备懒加载 BGE-Reranker-v2-m3 CrossEncoder。
- `rerank(query, passages, top_k, device=None)`：对 `[query, passage]` pair 打分，返回 `(原始下标, 分数)`。

### `retrieval/semantic_search.py`

- `_load()`：懒加载 `faiss.index`、`chunk_ids.json`、`chunk_store.pkl`。
- `semantic_search(query, top_k=SEMANTIC_TOP_K, rerank_top_k=RERANK_TOP_K)`：
  1. 用 BGE-M3 编码 query。
  2. 在 FAISS `IndexFlatIP` 上取 top-k。
  3. 根据 `chunk_ids` 找回文档。
  4. 用 BGE reranker 重新排序。
  5. 返回 top rerank results。

### `retrieval/keyword_search.py`

- `tokenize(text)`：中英文混合分词。中文用 jieba，英文/数字按正则切。
- `_load()`：加载 BM25、chunk ids 和 chunk store。
- `keyword_search(query, top_k=BM25_TOP_K)`：
  1. 分词。
  2. BM25 打分。
  3. 取正分 top-k。
  4. 如果结果超过 5 条，再用 reranker 取 top 5。

适用场景：公司名、财报字段、代码、具体实体等精确查询。

### `retrieval/read_chunk.py`

- `_load()`：加载 chunk store。
- `read_chunk(chunk_id)`：按 chunk_id 读取完整 chunk。返回列表是为了和其他 search 工具接口统一。

### `retrieval/graph_search.py`

- `_load_graph()`：加载 `knowledge_graph.json`。
- `_load_entity_embeddings()`：加载实体 embedding。
- `_load_chunk_store()`：加载 chunk store。
- `_match_entities(query, top_k=5)`：用 BGE-M3 编码 query，与实体 embedding 点积，找 top entity。
- `_bfs_collect_chunks(G, seed_entities, max_hops=2)`：从种子实体出发，在有向图上同时看出边和入边，按距离衰减收集相关 chunk。
- `graph_search(query, top_k)`：实体匹配 -> BFS 收集候选 chunk -> rerank -> 返回结果。

知识图谱的构建逻辑在 `scripts/build_knowledge_graph.py`，节点属性用 `mentions` 保存实体出现的 chunk，边上用 `chunk_id` 保存三元组来源。

### `retrieval/hybrid_search.py`

- `rrf_fuse(results_list, k=60)`：Reciprocal Rank Fusion，多路结果按 chunk_id 去重累积分数。
- `hybrid_fuse_and_rerank(query, results_list, top_k)`：先 RRF 融合，再取 top 15 做 CrossEncoder rerank。
- `multi_tool_search(query, tool_names, tool_registry, top_k)`：并行调用多个工具，再融合重排。

面试重点：hybrid 不是简单拼接，而是“多路并行召回、RRF 去重融合、CrossEncoder 精排”。

## 八、API 层

### `api/server.py`

FastAPI 演示接口。

- `QueryRequest`：请求体，包含 `question` 和 `verbose`。
- `QueryResponse`：响应体，包含 answer、query_type、iteration_count、tool_calls、evidence summary、trace、latency。
- `query_endpoint(req)`：调用 `agents.graph.run_query()`，把完整 state 压缩成可读的接口响应。
- `health()`：健康检查。

运行方式：

```bash
python api/server.py
```

接口：

```text
POST /query
GET  /health
```

## 九、评测层

### `evaluation/metrics.py`

- `_normalize(text)`：小写、去标点、去英文冠词、合并空白。
- `exact_match(prediction, gold, aliases=None)`：EM，支持 aliases。
- `f1_score(prediction, gold, aliases=None)`：token-level F1，多个 aliases 取最大。
- `_f1_single()`：单个 gold 的 F1。
- `CostTracker`：记录 tool calls、iterations、latency，并输出均值汇总。

### `evaluation/hop_aware_eval.py`

用于诊断多跳 RAG 的失败类型。

- `compute_hop_recall(state, qa_item)`：gold hop 的 `doc_chunk_id` 有多少被 evidence 召回。
- `diagnose_failure(state, qa_item)`：输出：
  - `hop_recall`
  - `premature_collapse`：hop recall 低且一轮就结束。
  - `over_extension`：工具调用远超 hop 数。
  - `step_alignment`：evidence step 数与 hop 数的比例。
- `aggregate_diagnostics(diagnostics)`：按 overall 和 hop_count 聚合。

这部分能回答“你的 Agent 哪里错了”而不只是报 EM/F1。

### `evaluation/llm_judge.py`

提供三类 LLM-as-a-Judge：

- `judge_answer_correctness(question, prediction, gold)`：答案正确性。
- `judge_faithfulness(question, answer, evidence)`：答案是否被检索证据支持。
- `judge_context_precision(question, evidence, gold_docs)`：检索证据是否覆盖 gold docs。

内部有中英文 prompt，并通过 `_truncate()` 限制 evidence 长度，防止 judge 输入过长。

### `evaluation/run_eval.py`

标准评测主入口。

- `load_qa_pairs(subset=None, data_dir=None)`：加载 `qa_pairs.json`。
- `_get_checkpoint_path()` 和 `_load_checkpoint()`：断点续跑。
- `run_full_eval(...)`：
  1. 可覆盖模型、语言、索引目录。
  2. 如果有 CUDA，预加载 embedder/reranker，并 monkeypatch 默认 device。
  3. 构建 LangGraph app。
  4. 按样本运行 Agent。
  5. 计算 EM/F1、hop-aware 诊断、latency 和 evidence_text。
  6. 可选 LLM Judge。
  7. 写 checkpoint 和最终 JSON。

### `evaluation/ablation.py`

做系统消融。

内置配置：

- `full_system`：keyword + semantic + read_chunk + verifier。
- `no_keyword`：去掉 BM25。
- `no_read_chunk`：去掉 chunk 读取。
- `no_verifier`：去掉证据验证重规划。
- `semantic_only`：朴素语义 RAG 基线。

函数：

- `run_ablation()`：跑一个配置。
- `run_all_ablations()`：跑多个配置并保存汇总。
- `print_comparison_table()`：打印 EM/F1/hop/calls/latency 对比。

## 十一、多跳 QA 合成链路

### `scripts/synthesis_llm.py`

合成脚本的 LLM 调用封装。

- `init_concurrency(max_concurrent)`：设置并发信号量。
- `get_stats()` / `reset_stats()` / `_record_call()`：统计调用次数、错误和延迟。
- `_extract_json(text)`：容错提取 JSON。
- `llm_call(prompt, model, temperature, timeout)`：统一走 `llm.client.get_from_llm()`。
- `llm_call_with_retry(..., return_json=False)`：带重试，可要求 JSON。
- `llm_judge(...)`：答案等价性评分，返回 `avg_score`。

### `scripts/gen_seed_qa.py`

从单个 chunk 生成原子 QA。

- `_tokens()`、`normalize_answer()`：文本标准化。
- `filter_qa(question, answer, chunk_text)`：过滤答案过长、答案泄露、问题引用文档、复合答案等低质量 QA。
- `process_chunk(chunk, prompts, model, gen_qa_num)`：
  1. 跳过短 chunk。
  2. 调 LLM 生成 QA。
  3. 去重和过滤。
  4. 对答案做 refine。
  5. 输出 seed QA。
- `main()`：并发处理 corpus，支持 resume、采样、prompt 文件。

### `scripts/domain_multihop_synthesis.py`

这是项目数据合成的核心，目标是从 seed QA 自底向上扩展成 2/3/4-hop QA。

基础函数：

- `normalize_answer()`、`f1_score()`：用于过滤和验证。
- `simple_partial_presence()`：检查上一跳答案是否泄露到下一跳问题中。
- `_years()`：年份一致性检查。

`_GPUModelPool`：

- 在多个 GPU 上加载 BGE-M3 和 reranker。
- 用 Queue 管理并发 slot。
- `acquire()` 获取模型，`release()` 归还。

`DomainRetriever`：

- 加载 FAISS、BM25、chunk_store。
- `search()`：从 GPU 池取模型。
- `_search_impl()`：
  1. FAISS 召回。
  2. BM25 召回。
  3. RRF 融合。
  4. reranker 精排。
  5. 可按 `seed_title` 做跨文档优先，避免只在同一公司同一文档附近打转。

`DomainMultiHopPipeline`：

- `process_seed()`：从一个 seed 出发逐跳扩展，控制每层 quota 和每个 seed 最大输出。
- `_extend_one_hop()`：扩展一跳的核心。
  1. 排除已用 chunk 和同公司相邻页 chunk，减少表格近邻造成的伪多跳。
  2. 主查询用上一跳 final question，辅查询用上一跳 refined answer。
  3. 合并 semantic/BM25 和 graph 检索结果。
  4. 对候选文档生成新原子 QA。
  5. 过滤 QA。
  6. 合成多跳问题。
  7. 四重验证，通过才保留。
- `_qa_depth_score()`：优先保留更有推理深度、少查表味的问题。
- `_filter_generated_qas()`：过滤重复、答案泄露、复合答案、引用文档等。
- `_merge_multihop()`：调用 LLM 把已有 hop 和新 QA 合并成一个多跳问题。
- `_verify_and_build()`：最重要的质量门控。
  - 前置过滤：comparison 泛化答案、inference 答案不一致、问题没变长、年份缺失、中间答案泄露。
  - refine final answer。
  - 生成 answer aliases。
  - 语义检查：合成问题和答案是否符合两个文档。
  - 推理检查：不给文档时模型不应答出答案。
  - 单文档检查：只给 N-1 个文档时模型不应答出答案。
  - 全文档检查：给全部支撑文档时模型应答出答案。
  - 构建新的 hop 数据。
- `_flatten_result()`：把内部 hop 结构转成最终 JSONL。
- `main()`：加载 prompts、corpus、seeds，初始化检索器和 pipeline，并发合成。

面试重点：这不是随便让 LLM 编多跳题，而是用“检索候选文档 + 多跳合成 + 四重验证”控制质量。

### `scripts/clean_synthesis.py`

清洗合成 QA。

- `_is_numeric_answer()`：识别纯数字/金额答案。
- `is_trivial_hop()`：判断是否是简单查表问题。
- `filter_trivial_last_hop()`：过滤 3hop+ 的最后一跳查表。
- `filter_all_trivial()`：过滤所有 hop 都查表。
- `dedup_by_question()`：按问题前缀去重。
- `dedup_by_chunk_overlap()`：按 chunk 集合重叠去重。
- `print_stats()`：统计类型分布、alias、trivial 比例。

### `scripts/judge_synthesis.py`

对合成 QA 做 LLM Judge 清洗。

- `build_hop_chain(qa, corpus_lookup)`：把每个 hop 的问题、答案、证据 chunk 摘要拼成 judge 输入。
- `judge_one()`：从 answer correctness、multihop necessity、question clarity 三个维度打分。
- `filter_by_score()`：按总分过滤。
- `main()`：支持并发 judge、resume、filter-only。

### `scripts/split_train_test.py`

训练/测试集划分。

- `group_by_seed_chunk()`：按第一跳 seed chunk 分组，防止同源 QA 进到训练和测试两边。
- `stratified_split()`：按 hop_count 和 qa_type 分层抽组。
- `print_distribution()`：输出 hop/type/judge/seed 分布。
- `check_leakage()`：检查 seed chunk 泄露。

### `scripts/translate_qa.py`

把英文或混合金融 QA 翻译成中文。

- `_is_chinese()`：判断是否已是中文或纯数字符号。
- `translate_one()`：调用 judge 模型翻译 question/answer。
- `process_file()`：并发处理 JSON 文件。

### `scripts/gen_enhanced_aliases.py`

用 LLM 给答案扩充 aliases，例如中文翻译、无单位数字、简称、去括号形式。

- `_get_alias_prompt()`：中英文 prompt。
- `gen_aliases(qa)`：单条生成 aliases。
- `main()`：并发处理并写回。

### `scripts/annotate_search_tools.py`

为 QA 的每个 hop 标注哪些检索工具能召回目标 chunk。

- `search_keyword()`、`search_semantic()`、`search_graph()`：工具包装。
- `annotate_one()`：

## 十二、SFT 数据与训练

### `scripts/build_oracle_traces.py`

把已有 gold hops 转成 oracle trajectory。

- `build_oracle_trace(qa, corpus_map, use_zh=False)`：
  1. 构造 router trace。
  2. 每个 hop 变成 planner step。
  3. 每个 hop 的 gold chunk 变成 executor evidence。
  4. verifier 固定 sufficient。
  5. synthesizer 输出 gold answer。
  6. 返回完整 trace、plan、evidence、tool_calls。
- `main()`：读取 QA 和 corpus，检查 chunk 覆盖率，输出 `traces_oracle*.jsonl`。

### `scripts/trace_to_sft.py`

把 trace 转成 ReAct SFT 格式。

- `_get_system_prompt()`：中英文系统 prompt。
- `_build_evidence_map()`：按 step_id 索引 evidence。
- `_format_observation()`：把工具返回结果格式化成 `[chunk_id] text`。
- `_normalize_tool()`：单工具保持原名，多工具变成 `hybrid_search`。
- `_make_tool_args()`：构建 tool_call arguments。
- `to_react(trace_data, lang=None)`：
  1. system + user。
  2. 每个 plan step 生成 assistant `<think>...<tool_call>...`。
  3. 插入 tool observation。
  4. 最后 assistant 输出 `<answer>gold</answer>`。
- `main()`：筛选正确、低迭代、有 evidence 的 trace，输出 `sft_react.jsonl`。

### `scripts/convert_sft_to_llamafactory.py`

转换成 LLaMA-Factory 的 sharegpt 格式。

- `convert_react()`：用 Qwen3 tokenizer 的 `apply_chat_template(tools=...)` 生成和 GRPO rollout 一致的训练文本，再拆回 messages。
- `convert_plan_execute()`：直接搬已有 messages。
- `convert_function_call()`：把 function_call 和 function role 转成文本/tool role。
- `register_datasets()`：写入 `LLaMA-Factory/data/dataset_info.json`。

### `training/sft_zh_react.yaml`

LLaMA-Factory SFT 配置：

- base model：Qwen3-4B。
- stage：sft。
- finetuning：LoRA，`lora_rank=64`，`lora_target=all`。
- dataset：`financial_agent_zh_react`。
- template：`qwen3_nothink`。
- cutoff_len：2048。
- batch：每卡 2，梯度累积 8。
- learning rate：`1e-4`。
- epoch：3。
- eval：10% validation，step eval/save。

### `training/sft_pipeline.sh`

一站式 SFT 流水线：

1. 清理 GPU 残留。
2. 用 LLaMA-Factory 训练 LoRA。
3. merge 每个 checkpoint。
4. 启动 vLLM 评测每个 merged checkpoint。
5. 批量 LLM Judge。
6. 汇总结果。

注意：当前脚本里的 `YAML_PATH="training/qwen3_4b_sft_zh_func_call.yaml"`，但仓库里实际看到的是 `training/sft_zh_react.yaml`。如果要直接跑，需要确认脚本和文件名是否同步。

### `scripts/prepare_agentic_grpo_data.py`

### `training/tools/financial_search_tool.py`

这是 verl 的工具实现，核心是把本项目的检索工具包装成 `BaseTool`。

共享加载函数：

- `_call_embed()`：HTTP 调 retrieval server 的 `/embed`。
- `_call_rerank()`：HTTP 调 `/rerank`。
- `_load_chunk_store()`、`_load_bm25()`、`_load_faiss()`、`_load_graph()`、`_load_entity_embeddings()`：共享单例加载。
- `_tokenize()`：中英文分词。
- `_rrf_fuse()`：RRF 融合。

工具基类 `_BaseFinancialTool`：

- `__init__()`：读取 index_dir、top_k、max_text_len、retrieval_server_url。
- `_format_results()`：把检索结果格式化成模型可读的 `[chunk_id] text`。
- `_rerank()`：通过远程 reranker 对候选重排。
- `_encode()`：通过远程 embedder 编码。
- `create()`：为 verl 创建 tool instance。
- `execute()`：执行 `_search()`，返回 `ToolResponse`、工具奖励 0、metadata。
- `calc_reward()`：工具本身不加奖励，返回 0。
- `release()`：释放 instance。

四个工具类：

- `KeywordSearchTool`：BM25 检索。
- `SemanticSearchTool`：FAISS 召回 + 远程 rerank。
- `GraphSearchTool`：实体 embedding 匹配 + KG BFS + 远程 rerank。
- `HybridSearchTool`：内部组合 keyword/semantic/graph，并行执行、RRF、rerank。

### `training/reward_agentic_rag.py`

GRPO 奖励函数，入口是 `compute_score(solution_str, ground_truth, **kwargs)`。

奖励组成：

```text
hop_precision_recall  * 0.30
judge_faithfulness    * 0.25
judge_correctness     * 0.25
grounded_answer       * 0.10
format_bonus          * 0.10
- insufficient_search_penalty
```

辅助函数：

- `_get_judge_client()` / `_call_judge()`：调用 OpenAI 兼容 Judge。
- `_judge_correctness()`：答案正确性。
- `_judge_faithfulness()`：证据忠实性。
- `_normalize()`：兼容中文、全角字符、标点、数字和中英文空格。
- `_token_f1()` / `_best_f1()`：答案 F1 和 aliases 最大匹配。
- `_extract_answer()`：抽 `<answer>...</answer>`。
- `_extract_evidence()`：抽 evidence 文本。
- `_extract_retrieved_chunks()`：从模型轨迹中抽 `[chunk_id]`。
- `_check_tool_call_format()`：检查 `<tool_call>{"name":...,"arguments":...}` 格式。
- `_count_tool_calls()`：统计工具调用次数。
- `_grounded_answer_score()`：答案 token 有多少出现在 evidence。
- `_hop_precision_recall()`：retrieved chunks 和 gold chunks 的 F1。

`compute_score()` 的特殊逻辑：

- `</think>` 过多直接 0，防止模型格式崩坏。
- 有 `<answer>` 给 0.06，有合法 `<tool_call>` 给 0.04。
- hop_pr 为 0 时给 -0.05 硬惩罚。
- tool calls 少于 hop_count 时扣 0.05。
- 如果 EM 或 F1 已经很高，correctness 直接 1，减少 judge 调用成本。
- correctness 和 faithfulness 可以并行调用。

## 十四、Agentic 模型评测脚本

### `scripts/eval_agentic.py`

这个脚本不走 LangGraph，而是模拟 GRPO 训练时的多轮 tool_call 环境，更适合评测经过 SFT/GRPO 的模型是否学会自主调用工具。

核心函数：

- `_load_retrieval(index_dir, device)`：一次性加载 BM25、FAISS、KG、embedder、reranker。
- `_keyword_search()`、`_semantic_search()`、`_graph_search()`、`_hybrid_search()`：本地工具环境，返回字符串 observation。
- `_build_system_prompt_with_tools()`：手动构造 Qwen3/HerMes 风格工具 schema，避免依赖 vLLM 原生 tool calling。
- `run_agentic_single(client, model, question, max_turns, temperature)`：
  1. system + user。
  2. 模型输出 `<tool_call>` 时解析 JSON。
  3. 调本地工具。
  4. 把 `<tool_response>` 作为下一轮 user message。
  5. 没有 tool_call 时提取 `<answer>`。
- `main()`：并发跑 QA，按 EM/F1、turns、tool_calls、latency 和 hop 分组汇总。

这个脚本是“训练后模型是否真的会用工具”的关键验证。

## 十六、面试中要主动讲清楚的设计点

### 1. 为什么需要 Agentic RAG，而不是普通 RAG

金融问题通常不是单文档单事实。例如“比较 A 公司经营现金流变化幅度和 B 公司营业成本变化幅度哪个更大”，需要：

1. 找 A 公司对应指标。
2. 找 B 公司对应指标。
3. 做比较。
4. 输出比较实体或数值。

普通 RAG 一次检索容易漏掉某一跳；Agentic RAG 通过 Planner 拆解、Executor 多次检索、Verifier 检查证据充分性，更适合多跳。

### 2. 为什么同时有 BM25、Semantic、Graph、Hybrid

- BM25：精确字段、公司名、指标名、代码效果好。
- Semantic：语义描述、表述变化、长问题效果好。
- Graph：实体关系、多跳链路、公司与指标之间的关联。
- Hybrid：召回风险高的关键步骤用多工具融合，RRF 降低单工具偏差。

### 3. Verifier 的作用

Verifier 不是评测模块，而是运行时控制模块。它决定“证据够不够”，不够就把 feedback 交给 Planner 重规划。为了防止死循环，还用 evidence overlap 检测强制结束重复检索。

### 4. SFT 和 GRPO 的分工

- SFT：用 oracle trace 教模型工具调用格式和基本搜索策略，是行为克隆。
- GRPO：让模型在真实检索环境中自主 trial-and-error，reward 同时考察检索命中、答案正确、faithfulness、格式。

### 5. 为什么 GRPO reward 不只看最终答案

如果只看答案 EM/F1，模型可能猜答案或输出常见实体。当前 reward 加入：

- hop precision/recall：鼓励检索到 gold chunks。
- faithfulness：鼓励答案被 evidence 支持。
- grounded answer：鼓励答案 token 出现在证据中。
- format：保持工具调用协议。
- tool_calls penalty：避免少搜。

这让训练目标更贴近 Agentic RAG 的全过程。

## 十九、面试官可能追问的问题和回答抓手

### Q1：你的系统和普通 RAG 最大区别是什么？

普通 RAG 是一次检索一次生成。我的系统是一个可循环的 Agentic RAG：先判断问题复杂度，多跳问题由 Planner 拆成子查询 DAG，Executor 每步选择 BM25/semantic/KG/hybrid 工具检索，Verifier 判断 evidence 是否足够，不够则重规划，最后 Synthesizer 只基于 evidence 输出答案。

### Q2：如何保证多跳数据不是 LLM 胡编？

数据合成不是裸生成。流程是 seed QA -> 检索相关候选文档 -> 生成新原子 QA -> 合并多跳问题 -> 四重验证。四重验证包括语义检查、无文档不能答、缺一部分文档不能答、给全量文档必须能答。然后还有 trivial 过滤、去重、LLM Judge 打分和 train/test 防泄露划分。

### Q3：为什么 reward 要包括 hop precision/recall？

因为在多跳问答中“答对”不一定说明检索过程正确，模型可能猜中。hop precision/recall 直接约束模型必须召回 gold chunks，同时 precision 惩罚引入过多无关 chunk。这样能让 GRPO 优化检索行为，而不只是优化最终文本。

### Q4：SFT 数据和 GRPO rollout 如何保持格式一致？

SFT 的 `trace_to_sft.py` 生成 Qwen3 原生 tool calling messages 和 tools schema；`convert_sft_to_llamafactory.py` 用 tokenizer `apply_chat_template(tools=...)` 生成训练文本。GRPO 的 tool config 也使用相同工具名和 schema，并设置 hermes 格式。这减少了 SFT 和 RL 阶段的格式偏移。

### Q5：知识图谱检索怎么做？

先通过 LLM 从 chunk 抽三元组，NetworkX 构建有向多重图，节点保存出现该实体的 chunk，边保存 relation 和 chunk_id。查询时先用 BGE-M3 把 query 匹配到相关实体，再从这些实体做 BFS，按距离衰减收集 chunk，最后用 reranker 精排。

### Q6：系统失败时怎么诊断？

除了 EM/F1，项目有 hop-aware 诊断：计算 gold hop chunks 的召回率，判断 premature collapse、over extension 和 step alignment。还可以用 LLM Judge 评答案正确性、faithfulness 和 context precision，把错误拆成“没检到、检到了但没用好、答案不忠实”。
